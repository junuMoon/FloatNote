Low-Fi Cycle

FloatNote should feel like a thin layer over work, not a second app.

This low-fi pass validates one thing first: whether the floating note window keeps attention on the body while still exposing note position, settings access, and quiet metadata at the bottom.

  • Hidden
  • Open on global shortcut
  • Resume last note
  • Write immediately
  • Navigate by creation order
  • Close back to work
Validation
  • Body owns most of the window.
  • Top bar stays thin and utility-like.
  • `Created` and `Updated` remain always visible but quiet.
  • New note creation reads as a navigation outcome, not a separate flow.
  • Settings stay small enough to feel secondary.

Screen 03

First-note boundary

When the user tries to move further left from the oldest note, the screen should stay put. A light edge response is enough. Anything heavier would interrupt the “quick note” rhythm.

  • No modal, toast, or warning sentence.
  • No layout shift. The note stays stable in place.
  • The feedback should confirm the edge without feeling punitive.

Screen 05

Settings sheet

Settings should feel like a short interruption, not another workspace. The visual weight stays lower than the main note window, and the ordering reflects the real priority: invoke key first, navigation second, size third.

  • Shortcut management stays grouped and obvious.
  • There is no broad preference taxonomy yet.
  • This can open as a sheet or a small independent window on macOS.

What this cycle locks in

  • Single-window note interaction stays intact.
  • The top bar remains utility-thin rather than editor-heavy.
  • Footer metadata is permanently visible.
  • New note creation is visually the same flow as note navigation.
  • Onboarding and settings stay secondary to writing.

What stays deliberately unresolved

  • Default previous/next note shortcuts.
  • 2nd-stage inline markdown rendering behavior.
  • Visual identity, color system, and motion language.
