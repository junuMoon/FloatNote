# FloatNote 기획 문서

## 1. 제품 한 줄 정의

`FloatNote`는 전역 단축키로 즉시 떠오르는 `floating document layer`다.  
메모 관리 UI보다 `바로 쓰기`를 앞세우고, 마크다운 문법은 같은 본문 표면 안에서 조용히 살아나야 한다.

## 2. 문제 정의

- 작업 중 짧은 메모를 남기기 위해 앱 전환과 창 정리가 필요하다.
- 많은 메모 앱은 기록보다 `관리 UI`가 먼저 보인다.
- 유틸성 도구는 기능이 많아질수록 오히려 첫 입력까지의 마찰이 커진다.

## 3. 제품 목표

- 기본 전역 단축키 `Control + A`로 어디서든 창을 열고 닫을 수 있다.
- 전역 단축키는 사용자가 설정에서 변경할 수 있다.
- 창은 현재 작업 위에 플로팅되지만, 메모보다 먼저 보이면 안 된다.
- 마지막으로 본 노트를 바로 연다.
- 본문은 마크다운 문법으로 작성되고, 같은 본문 표면 안에서 서식이 반영된다.
- 노트는 생성 순서 기준으로 왼쪽이 오래된 노트, 오른쪽이 더 새로운 노트다.
- 마지막 노트에서 오른쪽으로 이동하면 새 노트를 만든다.
- 하단에는 `Created`, `Updated`가 항상 보이지만 본문보다 조용해야 한다.

## 4. 제품 비목표

- 목록 화면, 대시보드, 사이드바를 만들지 않는다.
- 폴더, 태그, 프로젝트 구조를 초기 버전에 넣지 않는다.
- 저장 상태를 상시 과하게 드러내는 UI를 만들지 않는다.
- 메인 화면을 설정 중심 툴바 앱처럼 만들지 않는다.

## 5. 핵심 사용자 시나리오

### 시나리오 A. 순간 기록

1. 사용자가 IDE, 브라우저, 터미널 중 하나를 보고 있다.
2. `Control + A`를 누른다.
3. FloatNote가 현재 작업 위에 뜬다.
4. 마지막으로 보던 노트가 열리고 커서는 이미 본문에 있다.
5. 사용자는 바로 입력한다.
6. `Escape` 또는 같은 전역 단축키로 창을 닫고 원래 작업으로 돌아간다.

### 시나리오 B. 바로 앞뒤 노트 확인

1. FloatNote가 열린 상태에서 이전 노트 이동 단축키를 누른다.
2. 더 오래된 노트가 같은 창 안에서 전환된다.
3. 다음 노트 이동 단축키를 누르면 더 새로운 노트로 돌아온다.
4. 목록이나 브라우징 화면은 거치지 않는다.

### 시나리오 C. 새 생각 분리

1. 현재 노트를 쓰다가 새로운 갈래의 생각이 생긴다.
2. 다음 노트 이동 단축키를 누른다.
3. 다음 노트가 있으면 이동하고, 마지막 노트라면 빈 새 노트를 만든다.
4. 사용자는 이어서 바로 입력한다.

## 6. 디자인 방향

### 핵심 방향

- `content-first`
- `invisible chrome`
- `floating document`

### 메인 창 원칙

- 본문이 화면의 주인공이어야 한다.
- 설명 라벨보다 실제 텍스트가 먼저 보여야 한다.
- 상단 크롬은 얇고 조용해야 한다.
- 본문을 다시 카드 안에 넣지 않는다.
- 저장 상태, 모드 이름, 기술 설명을 메인 화면에 상시 노출하지 않는다.

### 메인 창에서 피할 것

- `LAST VIEWED NOTE`, `LIVE MARKDOWN STYLING` 같은 설명 라벨
- 크고 무거운 원형 버튼 나열
- 상시 `SAVED` 배지
- 설정과 닫기 버튼이 본문보다 먼저 보이는 구조
- 문서 위에 또 하나의 에디터 박스를 얹는 구조

## 7. 기능 요구사항

### 필수 기능

- 전역 단축키로 열기/닫기
  - 기본값: `Control + A`
  - 사용자가 설정에서 변경 가능
- 플로팅 창
  - 다른 앱 위에 뜨되, 빠르게 닫히고 포커스 복귀가 자연스러워야 함
- 노트 생성/조회/수정
- 마지막으로 본 노트 다시 열기
- 생성 순서 기반 이전/다음 이동
- 첫 노트에서 왼쪽 이동 시 현재 노트 유지
- 마지막 노트에서 오른쪽 이동 시 새 노트 생성
- 자동 저장
  - 입력 중 debounce 저장
  - 노트 이동 시 즉시 저장
- 하단 `Created`, `Updated` 표시
- 같은 본문 안에서의 인라인 마크다운 스타일링

### 메인 화면 요구사항

- 상단 고정 액션은 최소 개수만 유지한다.
- 메인 화면에서 제목 입력칸과 본문 입력칸을 분리하지 않는다.
- 키보드가 1순위이고 마우스 액션은 보조 수단이다.
- 설정은 별도 무거운 환경설정 창보다 가벼운 팝오버나 시트에 가깝게 유지한다.

## 8. UX 초안

### 창 구조

- 상단: 최소 크롬, 현재 위치, 최소 액션
- 본문: 문서형 입력 영역
- 하단: `Created`, `Updated`

### 입력 방식

- 창이 뜨면 커서가 바로 본문에 있어야 한다.
- 본문은 넓은 캔버스가 아니라 `읽기 편한 문서 폭`을 유지해야 한다.
- 마크다운 서식은 화려한 렌더러보다 읽기 흐름을 돕는 정도로 반영한다.

### 상태 표시

- 저장 상태는 상시 배지가 아니라 필요 시 짧게 드러나는 수준이 적절하다.
- 앱 이름은 브랜딩용으로만 남고, 본문보다 앞서면 안 된다.

### 키보드 동작

- `Control + A`: 기본 열기/닫기
- 이전 노트 이동 단축키: 기본값 추후 확정
- 다음 노트 이동 단축키: 기본값 추후 확정
- `Escape`: 창 닫기

## 9. 데이터 모델 초안

```ts
type Note = {
  id: string;
  body: string;
  createdAt: string;
  updatedAt: string;
};
```

### 정렬 규칙

- 기본 정렬은 생성 순서 기준
- 앱을 열면 마지막으로 본 노트를 연다
- 왼쪽은 더 오래된 노트
- 오른쪽은 더 새로운 노트
- 첫 노트의 왼쪽은 no-op
- 마지막 노트의 오른쪽은 새 노트 생성

### 이후 검토

- SQLite
- Markdown export

## 12. 리스크

- `Control + A`는 환경에 따라 충돌 가능성이 높다.
- 전역 단축키 등록은 macOS 권한 및 시스템 충돌 이슈가 있다.
- 크롬을 줄이다 보면 액션 발견성이 떨어질 수 있다.
- 인라인 마크다운 스타일링이 과해지면 오히려 입력 안정성을 해칠 수 있다.

## 13. MVP 완료 기준

- 전역 단축키로 창이 뜬다.
- 앱을 열면 마지막으로 본 노트가 열린다.
- 창이 뜨자마자 본문에 입력할 수 있다.
- 생성 순서 기반 좌우 이동이 가능하다.
- 마지막 노트 오른쪽 이동에서 새 노트가 생성된다.
- 하단에 `Created`, `Updated`가 보인다.
- 메인 화면이 본문보다 크롬을 더 강조하지 않는다.

## 14. 다음 단계

1. content-first 메인 창 레이아웃으로 리디자인한다.
2. 새 와이어프레임과 low-fi 시안을 만든다.
3. 실제 앱 UI를 그 방향으로 다시 구현한다.
4. 인라인 마크다운 스타일링을 읽기 중심으로 다듬는다.
