# FloatNote UX 동선 명세

## 1. 설계 관점

이번 문서는 개발 구현보다 사용자 행동 동선에 집중한다.  
FloatNote는 "빠르게 떠서, 생각을 붙잡고, 빠르게 사라지는" 도구여야 한다.

핵심은 기능 수보다 흐름이다.

- 사용자는 "노트를 관리"하려고 앱을 여는 것이 아니다.
- 사용자는 "지금 떠오른 생각을 놓치지 않기 위해" 앱을 순간적으로 호출한다.
- 따라서 모든 UI는 "불러오기", "기록", "이동", "닫기" 4단계의 마찰을 줄이는 방향으로 설계해야 한다.

## 2. 제품 원칙

### 원칙 1. 호출은 즉각적이어야 한다

- 단축키를 누른 뒤 창이 바로 떠야 한다.
- 사용자는 앱이 실행되는 과정을 느끼면 안 된다.
- 기본 단축키는 `Control + A`지만, 어디까지나 기본값일 뿐이며 사용자가 바꿀 수 있어야 한다.

### 원칙 2. 첫 화면은 항상 "입력 가능한 상태"여야 한다

- 창이 뜨면 커서는 바로 본문 입력 영역에 있어야 한다.
- 사용자가 추가 클릭 없이 바로 타이핑할 수 있어야 한다.

### 원칙 3. 쓰는 순간 읽혀야 한다

- 최종 목표는 읽기와 쓰기가 분리되지 않는 것이다.
- 사용자는 마크다운 문법으로 입력하고, 결과는 노션처럼 본문 안에서 바로 서식으로 반영되는 것이 이상적이다.
- 다만 MVP에서는 입력 안정성을 우선하기 위해 원문 입력만 먼저 제공할 수 있다.

### 원칙 4. 노트 이동은 "탐색"이 아니라 "흐름 유지"여야 한다

- 이전/다음 이동은 파일 브라우징처럼 느껴지면 안 된다.
- 사용자는 지금 보고 있는 노트, 더 오래된 노트, 더 새로운 노트 사이를 자연스럽게 이동해야 한다.

### 원칙 5. 메타데이터는 보이되, 방해되면 안 된다

- `Created`, `Updated`는 하단 고정 영역에 항상 보인다.
- 하지만 본문보다 더 눈에 띄면 안 된다.

## 3. 사용자의 기본 행동 루프

FloatNote의 핵심 루프는 아래 5단계다.

1. 떠오른 생각이 생긴다.
2. 전역 단축키로 FloatNote를 호출한다.
3. 바로 입력한다.
4. 필요하면 좌우로 이전 노트 또는 새 노트로 이동한다.
5. 닫고 원래 작업으로 복귀한다.

이 루프가 2초 안에 진입 가능해야 이 앱은 의미가 있다.

## 4. 주요 상태 정의

사용자 입장에서 FloatNote는 아래 상태들로 이해할 수 있다.

### 상태 A. 숨김

- 앱 창이 보이지 않는다.
- 사용자는 원래 작업을 하고 있다.
- 이 상태에서 전역 단축키를 누르면 FloatNote가 열린다.

### 상태 B. 즉시 입력

- 창이 뜨고 본문에 포커스가 있다.
- 마지막으로 보던 노트가 열린다.
- 사용자는 바로 메모를 쓸 수 있다.
- 가장 중요한 상태다.

### 상태 C. 서식 확인

- 사용자가 방금 쓴 내용을 읽는다.
- 인라인 렌더링이 있는 단계에서는 링크, 목록, 코드블록, 체크리스트를 본문 안에서 바로 확인한다.
- MVP 단계에서는 이 상태를 약하게 가져가고, 입력 상태를 우선한다.

### 상태 D. 이동

- 사용자가 이전/다음 노트 이동 단축키로 노트를 넘긴다.
- 이동은 페이지 전환처럼 느껴져야 하고, 목록 화면으로 빠져서는 안 된다.

### 상태 E. 복귀

- 사용자가 `Escape` 또는 전역 단축키로 창을 닫는다.
- 포커스는 원래 작업하던 앱으로 자연스럽게 돌아간다.

## 5. 핵심 동선 설계

### 동선 1. 순간 기록

이 동선은 FloatNote의 가장 중요한 사용 시나리오다.

1. 사용자는 브라우저, IDE, 터미널 중 하나를 보고 있다.
2. 어떤 생각, 명령어, 문장 초안, 링크 메모가 필요해진다.
3. 사용자는 기본 단축키 `Control + A`를 누른다.
4. FloatNote 창이 현재 화면 위에 뜬다.
5. 마지막으로 보던 노트가 열리고, 커서는 이미 본문에 있다.
6. 사용자는 즉시 입력한다.
7. 앱은 자동 저장한다.
8. 사용자는 `Escape` 또는 같은 전역 단축키로 닫는다.
9. 바로 이전 작업으로 돌아간다.

이때 가장 중요하게 지켜야 할 것은 두 가지다.

- 창이 뜨자마자 입력 가능해야 한다.
- 닫았을 때 원래 작업 흐름이 깨지지 않아야 한다.

### 동선 2. 이전 생성 노트 보기

1. 사용자가 FloatNote를 연다.
2. 현재 노트가 열려 있다.
3. 이전 노트 이동 단축키를 누른다.
4. 더 오래된 노트가 같은 창 안에서 전환된다.
5. 사용자는 내용을 참고하거나 일부 수정한다.
6. 다음 노트 이동 단축키로 방금 노트로 돌아온다.

여기서 중요한 점은 "목록 보기"를 거치지 않는 것이다.  
이 사용자는 정리하고 싶은 게 아니라 생성 순서상 인접한 노트를 바로 확인하고 싶다.

첫 노트에서 더 왼쪽으로 가려 하면 이동은 일어나지 않는다.  
이때는 과한 경고 대신, 가벼운 경계 피드백만 주는 편이 좋다.

### 동선 3. 생각을 쪼개서 이어쓰기

1. 사용자가 한 노트를 작성 중이다.
2. 관련 있지만 별도의 메모로 분리하고 싶은 생각이 생긴다.
3. 다음 노트 이동 단축키를 누른다.
4. 다음 노트가 있으면 그 노트로 이동한다.
5. 마지막 노트였다면 새 빈 노트가 열린다.
6. 사용자는 이어서 작성한다.

이 흐름에서 새 노트 생성은 버튼보다 이동의 연장선으로 느껴져야 한다.

### 동선 4. 읽기 중심 점검

1. 사용자가 작성한 마크다운을 눈으로 점검하고 싶다.
2. 인라인 렌더링이 있는 단계에서는 본문 안에서 서식이 반영된 결과를 확인한다.
3. 포맷이 이상하면 바로 수정한다.
4. 수정 결과가 곧바로 반영된다.

여기서 중요한 것은 "편집"과 "결과 확인"이 따로 놀지 않는다는 점이다.

## 6. 첫 화면의 구조

FloatNote는 처음 뜰 때 아래 순서를 따라 시선이 흐르는 것이 좋다.

1. 본문
2. 현재 노트 상태
3. 하단 메타데이터

즉, 화면의 주인공은 항상 본문이다.

### 권장 레이아웃

- 중앙 대영역: 마크다운 입력 본문
- 상단 얇은 상태영역: 앱 이름 또는 현재 노트 위치
- 하단 얇은 바: `Created`, `Updated`

### 피해야 할 것

- 상단 툴바가 지나치게 두꺼운 구조
- 버튼이 많은 문서 에디터 느낌
- 노트 앱보다 CMS처럼 보이는 정보량
- 별도 미리보기 패널이 기본 구조처럼 굳어지는 것

## 7. 입력 경험 설계

### 입력 시작

- 창이 뜰 때 기존 선택 상태나 placeholder보다 실제 커서가 더 중요하다.
- 빈 노트라면 사용자가 바로 첫 줄을 쓸 수 있어야 한다.
- 기존 노트라면 마지막 커서 위치를 기억하는 것이 이상적이다.

### 저장 피드백

- 저장은 자동이어야 한다.
- 저장 때문에 사용자가 흐름을 멈추면 안 된다.
- 별도의 큰 성공 메시지보다, 조용한 상태 표시가 적합하다.

### 마크다운 지원 범위

인라인 렌더링 단계에서는 아래만 확실히 좋아야 한다.

- 제목
- 리스트
- 체크리스트
- 코드블록
- 인라인 코드
- 링크
- 인용문

이 범위를 잘 만드는 것이, 많은 기능을 대충 넣는 것보다 낫다.

## 8. 노트 이동 경험 설계

### 좌우 이동의 의미

- `Left`는 "더 오래된 노트로 가기"
- `Right`는 "더 새로운 노트로 가기"

즉, 좌우 이동은 생성 순서의 시간축으로 느껴져야 한다.

### 양 끝 경계

- 첫 노트에서 `Left`는 no-op
- 마지막 노트에서 `Right`는 새 노트 생성
- 경계에서는 모달 대신 짧은 시각적 피드백만 사용한다.

### 전환 애니메이션

- 너무 화려할 필요는 없다.
- 짧고 얕은 슬라이드 정도면 충분하다.
- 중요한 것은 방향감이다.

### 마지막 노트의 오른쪽

- 마지막에서 다음 이동을 누르면 새 노트 생성
- 이때 "새 노트를 만들었습니다"라는 모달은 필요 없다.
- 사용자는 이미 새 캔버스가 열린 것을 보면 이해한다.

## 9. 닫기 경험 설계

닫기는 열기만큼 중요하다.

- `Escape`는 가장 빠른 종료 수단이어야 한다.
- 전역 단축키를 다시 눌러도 닫혀야 한다.
- 닫히는 순간 현재 내용은 저장되어 있어야 한다.
- 닫은 뒤에는 원래 보던 앱으로 시선과 포커스가 되돌아가야 한다.

사용자는 "앱을 종료했다"는 느낌보다 "잠깐 메모를 끼워 넣고 돌아왔다"는 느낌을 받아야 한다.

## 10. 설정 경험 설계

설정은 자주 쓰는 화면이 아니지만 초반 만족도에 큰 영향을 준다.

### 반드시 있어야 하는 설정

- 전역 단축키 변경
- 기본 창 크기

### 핫키 변경 UX

- 첫 실행 또는 설정 화면에서 현재 단축키를 보여준다.
- 사용자가 새 키 조합을 입력하면 즉시 충돌 여부를 검사한다.
- 충돌 시에는 실패만 알리지 말고 대체 조합 후보를 보여준다.

## 11. 첫 실행 경험

첫 실행에서는 앱의 모든 기능을 설명할 필요가 없다.

사용자가 알아야 할 것은 세 가지뿐이다.

1. 어떻게 띄우는가
2. 어떻게 쓰는가
3. 어떻게 넘기는가

따라서 첫 실행 안내는 짧아야 한다.

- 기본 단축키 안내
- 핫키 변경 가능 안내
- 좌우로 노트 이동 가능 안내

## 12. UX 관점에서의 우선순위

MVP에서 가장 중요한 것은 아래 순서다.

1. 뜨자마자 입력 가능
2. 닫자마자 원래 흐름으로 복귀 가능
3. 좌우 이동이 직관적일 것
4. 마지막으로 보던 노트에서 바로 이어쓸 수 있을 것
5. 메타데이터가 조용히 유지될 것

## 13. 단계 구분

- 1단계 MVP는 입력 흐름을 완성하는 데 집중한다.
- 2단계에서 노션 스타일 인라인 마크다운 렌더링을 붙인다.
- 따라서 지금의 UI 구조는 별도 미리보기 패널 없이도 완결되어야 한다.

## 14. 한 문장으로 정리

FloatNote는 "메모 앱"이 아니라, 작업 흐름 위에 잠깐 떠올랐다가 사라지는 사고 보조 레이어처럼 동작해야 한다.
