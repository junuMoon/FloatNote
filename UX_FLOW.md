# FloatNote UX 동선 명세

## 1. 설계 관점

FloatNote는 `메모 앱`보다 `작업 위에 잠깐 뜨는 문서`에 가깝다.  
사용자는 노트를 관리하려고 이 앱을 여는 것이 아니라, 지금 떠오른 생각을 놓치지 않기 위해 순간적으로 호출한다.

따라서 모든 UX 판단은 아래 질문으로 수렴해야 한다.

- 창이 뜨자마자 바로 쓸 수 있는가
- 앱 크롬보다 본문이 먼저 보이는가
- 닫았을 때 원래 작업 흐름으로 매끈하게 돌아가는가

## 2. 제품 원칙

### 원칙 1. 호출은 즉각적이어야 한다

- 단축키를 누른 뒤 창이 바로 떠야 한다.
- 기본 단축키는 `Control + A`지만, 사용자가 바꿀 수 있어야 한다.

### 원칙 2. 첫 화면은 항상 입력 가능한 상태여야 한다

- 창이 뜨면 커서는 본문에 있다.
- 사용자는 추가 클릭 없이 바로 타이핑할 수 있다.

### 원칙 3. 본문이 크롬보다 앞서야 한다

- 사용자는 앱 UI를 읽으러 들어오는 것이 아니다.
- 화면에서 가장 먼저 읽히는 것은 실제 노트 텍스트여야 한다.
- 앱 이름, 상태 라벨, 기능 설명은 본문보다 앞서면 안 된다.

### 원칙 4. 쓰는 순간 읽혀야 한다

- 읽기와 쓰기는 같은 표면에서 일어나야 한다.
- 마크다운 문법은 별도 미리보기 없이 같은 본문 안에서 살아나야 한다.
- 렌더링은 과시용이 아니라 가독성 향상용이어야 한다.

### 원칙 5. 이동은 브라우징이 아니라 흐름 유지여야 한다

- 이전/다음 이동은 파일 탐색처럼 느껴지면 안 된다.
- 현재 노트의 바로 앞뒤를 넘겨 보는 동작처럼 느껴져야 한다.

### 원칙 6. 메타데이터는 보이되 속삭여야 한다

- `Created`, `Updated`는 하단 고정 영역에 있다.
- 하지만 본문보다 더 먼저 보이면 안 된다.

### 원칙 7. 설정과 설명은 본문 밖으로 물러서야 한다

- 메인 화면은 설정 화면처럼 보이면 안 된다.
- 첫 실행 안내와 설정은 작은 callout, popover, sheet 수준이 적절하다.

## 3. 사용자의 기본 행동 루프

1. 떠오른 생각이 생긴다.
2. 전역 단축키로 FloatNote를 호출한다.
3. 바로 입력한다.
4. 필요하면 이전/다음 노트로 이동한다.
5. 닫고 원래 작업으로 복귀한다.

이 루프는 가능한 한 짧아야 한다.

## 4. 주요 상태 정의

### 상태 A. 숨김

- 앱 창이 보이지 않는다.
- 사용자는 원래 작업을 하고 있다.

### 상태 B. 문서 포커스

- 창이 뜨고 본문에 포커스가 있다.
- 마지막으로 보던 노트가 열린다.
- 가장 중요한 상태다.

### 상태 C. 이동

- 이전/다음 이동 단축키로 인접한 노트로 넘어간다.
- 같은 창, 같은 문서 표면 안에서 넘어가야 한다.

### 상태 D. 가벼운 보조 UI

- 설정 또는 첫 실행 안내가 잠깐 나타난다.
- 본문을 완전히 대체하지 않는다.

### 상태 E. 복귀

- `Escape` 또는 전역 단축키 재입력으로 닫는다.
- 포커스는 원래 작업 앱으로 돌아간다.

## 5. 핵심 동선

### 동선 1. 순간 기록

1. 사용자는 브라우저, IDE, 터미널 중 하나를 보고 있다.
2. 메모가 필요해진다.
3. `Control + A`를 누른다.
4. FloatNote가 뜬다.
5. 마지막으로 보던 노트와 커서가 즉시 나타난다.
6. 사용자는 바로 쓴다.
7. 자동 저장된다.
8. `Escape` 또는 같은 단축키로 닫고 복귀한다.

여기서 중요한 것은 두 가지다.

- 창이 뜨자마자 입력 가능해야 한다.
- 닫은 뒤에 작업 흐름이 끊기지 않아야 한다.

### 동선 2. 이전 생성 노트 확인

1. 사용자가 FloatNote를 연다.
2. 현재 노트가 열려 있다.
3. 이전 노트 이동 단축키를 누른다.
4. 더 오래된 노트로 전환된다.
5. 내용을 참고하거나 수정한다.
6. 다음 노트 이동 단축키로 다시 돌아온다.

이 흐름에서는 목록 보기로 빠지지 않는 것이 중요하다.

### 동선 3. 생각 분리

1. 사용자가 노트를 작성 중이다.
2. 별도 메모로 분리할 생각이 생긴다.
3. 다음 노트 이동 단축키를 누른다.
4. 마지막 노트가 아니면 다음 노트로 간다.
5. 마지막 노트라면 새 노트가 열린다.
6. 사용자는 이어서 입력한다.

새 노트 생성은 버튼 클릭보다 이동의 연장선처럼 느껴져야 한다.

## 6. 첫 화면 구조

FloatNote는 처음 뜰 때 아래 순서로 읽혀야 한다.

1. 본문 첫 줄 또는 현재 커서 근처
2. 현재 노트의 위치 감각
3. 하단 메타데이터

즉, 화면의 주인공은 항상 본문이다.

### 권장 구조

- 상단: 매우 얇은 크롬, 현재 위치, 최소 액션
- 본문: 문서형 입력 영역
- 하단: `Created`, `Updated`

### 피해야 할 것

- 두꺼운 툴바
- 설명 라벨 나열
- 큰 버튼 무리
- 상시 저장 배지
- 에디터 안의 또 다른 카드형 에디터

## 7. 입력 경험 설계

### 입력 시작

- placeholder보다 실제 커서가 더 중요하다.
- 빈 노트면 사용자가 바로 첫 줄을 쓸 수 있어야 한다.
- 기존 노트면 마지막 커서 위치를 기억하는 것이 이상적이다.

### 문서 폭

- 본문은 너무 넓은 캔버스가 아니라 읽기 편한 폭을 유지해야 한다.
- 시선이 화면 전체를 가로지르지 않게 해야 한다.

### 저장 피드백

- 저장은 자동이어야 한다.
- 큰 성공 메시지는 필요 없다.
- 저장 상태는 짧게 드러나거나 아예 배경에서 처리되는 편이 낫다.

### 마크다운 지원 범위

초기에는 아래가 좋아야 한다.

- 제목
- 리스트
- 체크리스트
- 코드블록
- 인라인 코드
- 링크
- 인용문

## 8. 액션 디자인

### 키보드 우선

- 열기, 닫기, 이동은 키보드가 1순위다.
- 마우스 액션은 보조 수단이어야 한다.

### 상단 액션 수

- 메인 화면 상단 액션은 최소 개수만 유지한다.
- `이전`, `현재 위치`, `다음`, `설정` 정도면 충분하다.
- `저장`, `닫기`는 상시 독립 버튼으로 크게 드러낼 필요가 없다.

### 아이콘 원칙

- 아이콘만으로 의미가 분명할 때만 아이콘을 쓴다.
- 모호한 액션은 메인 바에서 빼고 설정이나 메뉴로 보낸다.

## 9. 노트 이동 경험

### 좌우 이동 의미

- `Left`는 더 오래된 노트
- `Right`는 더 새로운 노트

즉, 좌우 이동은 생성 순서의 시간축이다.

### 양 끝 경계

- 첫 노트에서 `Left`는 no-op
- 마지막 노트에서 `Right`는 새 노트 생성
- 경계에서는 모달 대신 짧은 시각 피드백만 쓴다

### 전환 감각

- 화려한 애니메이션은 필요 없다.
- 짧고 얕은 방향감이면 충분하다.

## 10. 닫기 경험

- `Escape`는 가장 빠른 종료 수단이다.
- 전역 단축키를 다시 눌러도 닫혀야 한다.
- 닫히는 순간 현재 내용은 저장되어 있어야 한다.
- 닫은 뒤에는 원래 작업으로 돌아간 느낌이 들어야 한다.

## 11. 설정 경험

설정은 자주 머무는 화면이 아니다.

### 반드시 있어야 하는 설정

- 전역 단축키 변경
- 이전/다음 이동 단축키 변경
- 기본 창 크기

### 권장 구조

- 무거운 Preferences 창보다 작은 popover나 시트가 낫다.
- 메인 화면을 완전히 대체하는 느낌은 피한다.

## 12. 첫 실행 경험

첫 실행에서 사용자가 알아야 할 것은 세 가지뿐이다.

1. 어떻게 띄우는가
2. 어떻게 바로 쓰는가
3. 어떻게 넘기는가

그래서 첫 실행 안내는 짧아야 한다.

- 기본 단축키 안내
- 바로 입력 가능하다는 점
- 이전/다음 이동 가능하다는 점

## 13. UX 우선순위

1. 뜨자마자 입력 가능
2. 본문이 UI보다 먼저 보일 것
3. 닫자마자 원래 흐름으로 복귀 가능
4. 좌우 이동이 직관적일 것
5. 메타데이터가 조용히 유지될 것

## 14. 한 문장으로 정리

FloatNote는 `기능을 설명하는 앱`이 아니라, 작업 흐름 위에 잠깐 떠서 생각을 붙잡는 `조용한 문서`처럼 동작해야 한다.
